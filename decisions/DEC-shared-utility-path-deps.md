# DEC-shared-utility-path-deps: Cross-component shared utilities live in `3-code/_common/<package>/` and are consumed via uv path-deps

**Status**: Active

**Category**: Convention

**Scope**: backend (`whatsorga-ingest`, `hermes-runtime`, `backlog-core`, `gbrain-bridge`, `kanban-sync`)

**Source**: [DEC-backend-stack-python-fastapi](DEC-backend-stack-python-fastapi.md), [DEC-hash-chain-over-payload-hash](DEC-hash-chain-over-payload-hash.md)

**Last updated**: 2026-04-29

## Context

`DEC-backend-stack-python-fastapi` settled the backend stack but explicitly deferred "the location of cross-component shared utilities (separate package vs. per-component vendoring) … to be settled when `TASK-canonical-json-helper` lands." That task is now landing.

Phase 2 introduces the first of five planned cross-cutting helpers:

| Helper | Triggering decision/req | Phase |
|---|---|---|
| `canonical_json` | `DEC-hash-chain-over-payload-hash` | Phase 2 |
| `bearer_auth` middleware | `DEC-service-auth-bearer-tokens` | Phase 2 |
| `idempotency` middleware | `DEC-idempotency-keys` | Phase 3 |
| `purpose_limitation` middleware | `REQ-COMP-purpose-limitation` | Phase 3 |
| `subject_ref_normalization` | `ASM-subject-reference-resolvable` | Phase 4 |

All five are imported into all five backend services. Three approaches were evaluated:

- **Vendoring** — duplicate the utility into each component, enforce byte-identity in CI. Estimated ~3.2k LOC of duplication across 5 helpers × 5 components when fully landed; every fix touches 5 files; CI gains 5 byte-identity jobs.
- **uv path-deps + Dockerfile context restructure (chosen)** — `3-code/_common/<package>/` siblings to per-component dirs; each component's `pyproject.toml` declares the helper as a path-dep; Dockerfiles use the repo root as build context and copy both the component dir and `_common/`. CI gains one test job per shared package.
- **uv workspaces** — single workspace at the repo root with all 6 components + `_common/` packages as members. Rejected: it enforces a single shared lockfile across the workspace, which conflicts with the per-component `uv.lock` pattern that all 6 skeletons already established (and which keeps each component's deploy surface minimal).

The component-isolation rule in `3-code/CLAUDE.code.md` forbids cross-component coupling. Shared utilities are not cross-component coupling — they are declared cross-cutting infrastructure consumed by every backend, so the rule needs a narrow carve-out for `_common/` to permit this pattern explicitly.

## Decision

Cross-component shared utilities live as **independent uv-managed Python packages** under `3-code/_common/<package>/`, siblings to the per-component directories. Each backend component declares its consumed helpers as **uv path-deps** in its `pyproject.toml`. Component Dockerfiles use the **repo root as the build context** and copy both the component dir and the consumed `_common/` packages.

Specifically:

1. **Package layout** — each shared package is a self-contained uv project at `3-code/_common/<package>/`:
   ```
   3-code/_common/<package>/
     pyproject.toml          # name, version, deps; tool config (ruff, mypy, pytest)
     uv.lock                 # committed, generated by `uv lock`
     .python-version         # "3.12"
     <package>/              # importable Python package (use the canonical kebab→snake conversion)
       __init__.py           # public surface
     tests/
       __init__.py
       test_*.py
     README.md               # purpose, public surface, change policy
   ```
   The `_common/` directory itself does not require a `CLAUDE.component.md` — it is not a runtime component. Each shared package is independently tested and versioned.

2. **Component consumption** — each backend component's `pyproject.toml` declares the helper under `[tool.uv.sources]` (path-dep, editable install for development):
   ```toml
   [project]
   dependencies = [
       # ...
       "<package-name>",
   ]

   [tool.uv.sources]
   <package-name> = { path = "../_common/<package>", editable = true }
   ```
   The component's `uv.lock` pins the path-dep by relative path. `uv sync --frozen` resolves correctly from the component directory.

3. **Dockerfile build context** — backend Dockerfiles use the **repository root** as build context (changed from `./3-code/<component>/`). The Dockerfile explicitly copies the component dir and every consumed `_common/` package:
   ```dockerfile
   FROM python:3.12-slim AS builder
   COPY --from=ghcr.io/astral-sh/uv:0.9.16 /uv /uvx /usr/local/bin/
   WORKDIR /build
   ENV UV_PROJECT_ENVIRONMENT=/opt/venv UV_LINK_MODE=copy UV_COMPILE_BYTECODE=1
   # Layout the path-dep tree so uv can resolve `../_common/<package>` from
   # the component dir at build time.
   COPY 3-code/_common /build/3-code/_common
   COPY 3-code/<component>/pyproject.toml 3-code/<component>/uv.lock /build/3-code/<component>/
   WORKDIR /build/3-code/<component>
   RUN uv sync --frozen --no-dev --no-install-project
   COPY 3-code/<component>/app /build/3-code/<component>/app
   RUN uv sync --frozen --no-dev
   # ...runtime stage unchanged...
   ```
   Each Dockerfile only copies the `_common/` packages it actually consumes — but for simplicity, all 5 backends copy all of `3-code/_common/` (the additional layer cost is small and avoids per-Dockerfile drift).

4. **docker-compose.yml** — every backend service's `build:` block uses `context: .` + `dockerfile: 3-code/<component>/Dockerfile`. The `cli` service is unchanged (CLI does not consume `_common/` helpers — it has no Python deps in `_common/`'s scope).

5. **CI** — each `_common/<package>/` gets its own per-package CI job mirroring the per-component pattern (`uv sync --frozen` → ruff → mypy → pytest). Component test jobs unchanged in shape; the path-dep is resolved via `uv sync --frozen` exactly as before because the shared package sits in a known relative location.

6. **Component-isolation carve-out** — the rule in `3-code/CLAUDE.code.md` is amended to permit `3-code/_common/<package>/` as the single sanctioned location for cross-cutting utilities consumed by multiple components.

## Enforcement

### Trigger conditions

- **Code phase**: any task that creates or modifies cross-component shared code, or that adds/removes a backend component's dependency on such code, must consult this decision.
- **Code phase**: any task creating or modifying a backend Dockerfile or that backend service's `build:` block in `docker-compose.yml` must follow the build-context contract above.

### Required patterns

- New shared utilities go under `3-code/_common/<package>/`. No other location is sanctioned.
- The shared package's `pyproject.toml` declares its own runtime + dev deps; it does **not** import from any backend component.
- A consuming backend component declares the helper under `[project] dependencies` and `[tool.uv.sources]` with a path-dep. The path is relative (`../_common/<package>`), editable for development.
- Backend Dockerfiles use repo-root build context and copy both `3-code/_common/` and `3-code/<component>/` into the builder.
- `docker-compose.yml` backend services use `build: { context: ., dockerfile: 3-code/<component>/Dockerfile }`.
- Each shared package gets its own CI test job named `_common-<package>-test` (e.g., `_common-canonical-json-test`).
- The `_common/` carve-out applies **only** to declared cross-cutting utilities. Two-component-only utilities are not eligible — if only two components share something, ship it in the calling component or refactor responsibilities.

### Required checks

1. `uv sync --frozen` from a backend component dir resolves the path-dep without network access.
2. `docker build -f 3-code/<component>/Dockerfile -t <component> .` (run from repo root) succeeds.
3. `docker compose build <service>` succeeds for every backend service.
4. Each `_common/<package>/` CI job passes ruff + mypy strict + pytest.
5. The component-isolation rule's `_common/` carve-out is documented in `3-code/CLAUDE.code.md`.

### Prohibited patterns

- Vendoring (copy-pasting) shared utilities into multiple components.
- Importing across component directories (e.g., `from whatsorga_ingest.app import …` inside `hermes-runtime`) — still forbidden by the unmodified part of the component-isolation rule.
- Placing shared code anywhere other than `3-code/_common/<package>/` (e.g., a top-level `lib/` or per-component `vendor/` dir).
- uv workspaces — single-workspace mode would force a shared lockfile across the 6 components, conflicting with the per-component `uv.lock` pattern this scaffold already follows.
- Publishing `_common/` packages to PyPI or any external index. They are local-source-only.

## Reconsider trigger

Revisit this decision if:

- A shared utility's consumers diverge so much that they need different versions (would force genuine semver and possibly an external publish step).
- The team grows and a workspace-wide lockfile becomes desirable for unified upgrades.
- A `_common/` package needs to be reused outside this repo (would force PyPI publishing or a private index).
