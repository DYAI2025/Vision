# DEC-backend-stack-python-fastapi: Python 3.12 + FastAPI as the uniform backend stack

**Status**: Active

**Category**: Convention

**Scope**: backend (`whatsorga-ingest`, `hermes-runtime`, `backlog-core`, `gbrain-bridge`, `kanban-sync`)

**Source**: n/a

**Last updated**: 2026-04-28

## Context

The implementation plan defines five backend components plus several cross-cutting utilities (canonical-JSON helper, bearer-auth middleware, idempotency middleware, purpose-limitation middleware, subject-reference normalization) consumed by every backend service. Without a single backend stack, each utility would have to be re-implemented per language, multiplying maintenance and divergence risk for a two-person team. The component-isolation rule in `3-code/CLAUDE.code.md` already forbids cross-component coupling, so the way these utilities are shared (vendoring vs. shared package) is a separate sub-decision; choosing one language is a prerequisite to either approach.

`3-code/CLAUDE.code.md` and each backend component's `CLAUDE.component.md` already flagged "Python likely" pending the first per-component skeleton task. This decision settles the choice.

## Decision

All five backend components use **Python 3.12** with the following uniform stack:

| Concern | Choice |
|---|---|
| Runtime | Python 3.12 (Docker base: `python:3.12-slim`) |
| HTTP framework | FastAPI + uvicorn (ASGI) |
| Validation / models | Pydantic v2 (already a FastAPI dep) |
| Postgres driver (where applicable) | asyncpg |
| Dependency / venv manager | `uv` |
| Test runner | pytest + httpx (`AsyncClient` / `TestClient`) |
| Linter / formatter | ruff |
| Type checker | mypy strict |

The CLI component (`cli/`) is **out of scope** for this decision; its language choice is deferred to `TASK-cli-skeleton`.

The location of cross-component shared utilities (separate package vs. per-component vendoring) is **not** decided here; it will be settled when `TASK-canonical-json-helper` lands.

## Enforcement

### Trigger conditions

- **Code phase**: any task that creates or modifies source code, build configuration, or test infrastructure inside `3-code/whatsorga-ingest/`, `3-code/hermes-runtime/`, `3-code/backlog-core/`, `3-code/gbrain-bridge/`, or `3-code/kanban-sync/`.

### Required patterns

- Per-component layout (rooted at `3-code/<component>/`):
  ```
  Dockerfile
  pyproject.toml          # runtime + dev deps + tool config (ruff, mypy, pytest)
  uv.lock                 # committed lockfile (generated by `uv lock`)
  .python-version         # contains "3.12" — read by uv
  app/                    # importable package
    __init__.py
    main.py               # FastAPI app instance + GET /v1/health
    ...
  tests/                  # pytest test root
    __init__.py
    test_*.py
  README.md               # build/run/test commands
  .dockerignore
  .gitignore              # excludes .venv/, __pycache__/, .pytest_cache/, etc.
  ```
- `pyproject.toml` declares runtime deps (`fastapi`, `uvicorn[standard]`, `pydantic`) and dev deps under `[dependency-groups]` `dev` (`pytest`, `httpx`, `ruff`, `mypy`).
- `pyproject.toml` configures `[tool.ruff]`, `[tool.mypy]` (strict mode), and `[tool.pytest.ini_options]`.
- Dockerfile is multi-stage: builder uses `uv` to install deps into a venv; runtime stage copies the venv into `python:3.12-slim` and runs uvicorn under a non-root user. Listens on port 8000.
- All services expose `GET /v1/health` returning `{"status": "ok|degraded|down", "version": "...", "checks": {...}}` per `2-design/api-design.md` §"Health and observability". No auth on `/v1/health`.

### Required checks

1. New / modified backend components must build with `docker build -t <component> 3-code/<component>/` (CI compose-validate covers the indirect path; a per-component `<component>-test` CI job runs `uv run pytest` against the component).
2. `uv lock --check` (or `uv sync --frozen` in CI) must succeed — committed lockfile is the source of truth.
3. `uv run ruff check` and `uv run mypy app` must pass before a component's task is `Done`.

### Prohibited patterns

- Mixing languages across the five backend components (e.g., Go for `backlog-core` only) without first superseding this decision per `decisions/PROCEDURES.md`.
- Skipping the per-component venv (`.venv/` at the component root). Do not install deps system-wide or share a venv across components.
- Importing across component directories (still subject to `3-code/CLAUDE.code.md` component-isolation rule). Cross-cutting utilities live where the eventual sub-decision says they live, not in arbitrary cross-imports.
- Using poetry, pipenv, or pip-tools for dependency management — `uv` is the only sanctioned tool.
- `uvicorn --reload` in production images (dev-only flag).
