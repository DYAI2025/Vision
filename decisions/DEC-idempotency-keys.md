# DEC-idempotency-keys: Mutation endpoints accept Idempotency-Key headers

**Status**: Active

**Category**: Convention

**Scope**: system-wide (mutation endpoints)

**Source**: [DEC-direct-http-between-services](DEC-direct-http-between-services.md), [REQ-F-proposal-pipeline](../1-spec/requirements/REQ-F-proposal-pipeline.md), [REQ-REL-event-replay-correctness](../1-spec/requirements/REQ-REL-event-replay-correctness.md)

**Last updated**: 2026-04-27

## Context

Inter-service HTTP is synchronous (per `DEC-direct-http-between-services`) with timeouts and retry-with-jitter for transient failures. Without an idempotency mechanism, a retry after a network failure or a timeout can double-apply a mutation: a `proposal.proposed` event committed twice produces two cards, two consent updates, two RTBF cascades. This is incompatible with `REQ-F-proposal-pipeline`'s "every mutation flows through propose → validate → apply" semantics and with `REQ-REL-event-replay-correctness`'s idempotence property.

Choices: idempotency keys carried as a header, idempotency keys carried as a query parameter, idempotency by content hash (server computes and dedupes), no idempotency mechanism.

## Decision

All **mutation endpoints** (`POST`, `PATCH`, `DELETE`) accept an `Idempotency-Key` HTTP header. The receiving service:

1. Looks up the key in its idempotency store (persistent for `backlog-core`, in-memory + persisted for `gbrain-bridge` and `kanban-sync`).
2. If the key is present and the original request completed successfully, returns the original response (status code + body) without re-applying the mutation.
3. If the key is present and the original request is still in flight (rare, but possible on rapid retry), returns `409 idempotency_in_flight` and the client retries after a short backoff.
4. If the key is present and the original request body differs from the current request, returns `409 idempotency_conflict` (catches client bugs).
5. If the key is absent, the mutation is processed normally — duplicates are not detected.

Idempotency keys are stored for **at least 24 hours** after the request completes. Beyond that, they may be evicted; clients should not retry with the same key after 24 hours.

Recommended key sources:

- For proposal-pipeline calls: the **`proposal_id`** itself (already unique per logical proposal).
- For consent-management calls: a **per-CLI-invocation UUID** generated by the CLI.
- For RTBF and export calls: a **per-request UUID** generated by the CLI.
- For ingestion: the **`event_id`** of the candidate event.

## Enforcement

### Trigger conditions

- **Design phase**: any new mutation endpoint declares whether it accepts `Idempotency-Key` (default: yes).
- **Code phase**: HTTP middleware on every mutation endpoint extracts the header, performs the lookup, and either returns the cached response or proceeds with the request.

### Required patterns

- The idempotency store keyed by `(calling_identity, idempotency_key)` so different services using the same UUID don't collide.
- Stored entry includes: response status, response body, request body hash (for conflict detection), `created_at`, `completed_at`.
- TTL: 24 hours minimum; a TTL-sweep job removes expired entries.
- Key length: arbitrary (UUIDs are recommended) but ≤ 256 bytes.
- The header name is exactly `Idempotency-Key` (canonical capitalization).

### Required checks

1. Unit test: identical request with identical `Idempotency-Key` returns the cached response without re-emitting events.
2. Unit test: request with the same `Idempotency-Key` but different body returns `409 idempotency_conflict`.
3. Unit test: request without the header is processed normally; subsequent identical request without header processes again (no dedup).
4. Soak test: 24-hour-old entries are evicted; a request with the now-evicted key processes as a new request.

### Prohibited patterns

- Returning a different status code on retry than on the original request (would be a contract violation).
- Storing request bodies (only their hashes) in the idempotency store — bodies may contain sensitive content; hashes are sufficient for conflict detection.
- Using the idempotency mechanism for queue-style "at-least-once" semantics — this is a deduplication mechanism, not a delivery guarantee.

## Reconsider trigger

Revisit this decision if:

- Idempotency-store storage cost grows significant (would prompt review of TTL or pruning policy).
- A mutation endpoint emerges that is genuinely unsafe to retry even with idempotency (would prompt a more sophisticated retry-policy mechanism — none anticipated at MVP).
- Multi-stage transactional mutations across services need stronger consistency than per-endpoint idempotency provides (would prompt a saga / outbox pattern, recorded as a separate decision).
